# KDNA Crypto Protocol v0.1

## Status: Draft

This document defines how .kdna files are encrypted, signed, licensed, and verified — the cryptographic infrastructure for KDNA as a tradeable judgment asset.

**Design principle:** KDNA encryption does not promise "uncopyable files." It promises legitimate purchase, authorized use, leak tracing, and managed revocation. The goal is to raise the cost of unauthorized use high enough to make honest purchase the rational choice.

---

## 1. Access Modes

Every KDNA domain declares one of three access modes. The mode determines the cryptographic treatment.

| Mode | Distribution | At Rest | At Load | Revocable | Watermark |
|------|:-----------:|:-------:|:-------:|:---------:|:---------:|
| `open` | Plaintext .kdna | Plaintext | Direct read | No | Optional |
| `licensed` | Encrypted .kdna.pkg | Encrypted | Local decrypt with license key | Yes | Required |
| `runtime` | Never distributed | Server-side only | API projection | Yes | Required |

**Open mode** requires no cryptography. This document focuses on `licensed` and `runtime`.

## 2. Key Architecture

KDNA uses a two-layer key model:

```
.kdna.enc (encrypted domain content)
    ↓
  CK (Content Key) — symmetric, encrypts the domain
    ↓
  LK (License Key) — asymmetric, authorizes access to CK
    ↓
  Buyer's private key — held on buyer's device, never transmitted
```

### 2.1 Content Key (CK)

- **Type:** AES-256-GCM symmetric key
- **Generated by:** Registry at publish time
- **One per domain version** — same CK for all buyers of the same domain@version
- **Stored:** Encrypted with each buyer's public key in their license file
- **Never exposed in plaintext** outside the Registry's secure environment and the buyer's local runtime

### 2.2 License Key (LK) — Buyer Identity

- **Type:** Ed25519 key pair
- **Generated by:** Buyer's local machine (`kdna identity init`)
- **Private key:** Stored at `~/.kdna/identity/kdna.key` (chmod 600)
- **Public key:** Uploaded to Registry at purchase time
- **Registry NEVER sees the private key**
- **One identity per machine/team** — can authorize multiple domains

### 2.3 Registry Signing Key (SK)

- **Type:** Ed25519 key pair
- **Held by:** KDNA Registry
- **Purpose:** Signs license files and package manifests
- **Public key:** Bundled with kdna CLI, verified against registry on install

## 3. Publishing Flow (Author → Registry)

```
Author creates domain (plaintext JSON files)
    ↓
kdna publish --access licensed
    ↓
1. Registry generates CK (random AES-256 key)
2. Registry encrypts .kdna content with CK → domain.kdna.enc
3. Registry computes SHA-256 hash of plaintext content → content_hash
4. Author signs content_hash with author's Ed25519 private key
5. Registry creates package manifest → manifest.json
6. Registry signs manifest with SK
    ↓
Package published to registry:
  silver-care.kdna.pkg
  ├── manifest.json          (signed by Registry)
  ├── domain.kdna.enc        (encrypted with CK)
  ├── license.policy.json    (access mode, terms, revocation policy)
  └── watermark.profile.json  (watermark configuration)
```

### 3.1 Package Manifest

```json
{
  "kdna_spec": "1.0-rc",
  "crypto_protocol": "0.1",
  "domain": "silver-care",
  "version": "1.0.0",
  "access": "licensed",
  "content_hash": "sha256:abc123...",
  "encryption": {
    "algorithm": "AES-256-GCM",
    "ck_id": "ck_silver-care_1.0.0"
  },
  "author_signature": "ed25519:def456...",
  "registry_signature": "ed25519:ghi789..."
}
```

## 4. Purchase Flow (Buyer → Registry → Local)

```
Buyer runs: kdna license activate silver-care
    ↓
1. If no identity exists: kdna identity init
   → Generates Ed25519 key pair
   → Stores kdna.key (private) and kdna.pub (public)
   → Derives buyer_id = SHA-256(kdna.pub)[0:16]
    ↓
2. Buyer sends to Registry:
   {
     "domain": "silver-care",
     "buyer_id": "abc123...",
     "public_key": "ed25519:jkl012...",
     "payment_token": "..."
   }
    ↓
3. Registry validates payment
4. Registry encrypts CK with buyer's public key → ck_encrypted
5. Registry creates license file → license.kdna.json
6. Registry signs license with SK
    ↓
7. Buyer receives license.kdna.json:
   {
     "domain": "silver-care",
     "version": "1.0.0",
     "buyer_id": "abc123...",
     "access": "licensed",
     "issued_at": "2026-06-01T00:00:00Z",
     "expires_at": null,
     "ck_encrypted": "base64:...",
     "ck_encryption": {
       "algorithm": "ECDH-ES+A256KW",
       "recipient_key": "ed25519:jkl012..."
     },
     "registry_signature": "ed25519:..."
   }
    ↓
8. License stored at ~/.kdna/licenses/silver-care.json
```

---

## 5. Load Flow (Runtime Decryption)

```
Agent triggers: kdna use silver-care
    ↓
1. Runtime reads license from ~/.kdna/licenses/silver-care.json
2. Verifies registry_signature against known Registry public key
3. Checks: not expired, not revoked, domain matches
4. Decrypts ck_encrypted with buyer's private key → CK
5. Decrypts domain.kdna.enc with CK → plaintext KDNA (in memory only)
6. Injects semantic watermark (buyer_id + timestamp)
7. Projects only task-relevant fragments (projector.js)
8. Loads fragments into agent context
9. Logs audit entry (domain, timestamp, task_type, call_count)
    ↓
Plaintext KDNA NEVER touches disk.
```

---

## 6. Revocation Flow

```
Registry revokes buyer's license
    ↓
Registry updates license status → revoked
kdna CLI periodically syncs (kdna license sync)
    ↓
Next load attempt:
  License status: revoked
  Runtime refuses to decrypt
  Audit log records revocation check
```

Offline grace period: 7 days. After 7 days without sync, license is considered expired until next sync.

---

## 7. Watermark Policy

Every `licensed` and `runtime` domain MUST include watermark injection.

| Mode | Watermark Content | Injection Point |
|------|------------------|----------------|
| `licensed` | buyer_id + license_id + timestamp | Encoded in response text (zero-width marker) |
| `runtime` | buyer_id + call_id + timestamp | Encoded in API response |

If a watermarked response appears publicly, Registry can:
1. Extract watermark → identify buyer
2. Issue warning
3. Revoke license if repeated

This is NOT DRM. It is **leak accountability** — the buyer knows their identity is embedded and traceable.

## 8. .kdna.pkg Format

The `.kdna.pkg` file is a tar archive (uncompressed) with a deterministic order:

```
silver-care-1.0.0.kdna.pkg
├── kdna.pkg.json          (package manifest — always first)
├── domain.kdna.enc        (encrypted domain content)
├── license.policy.json     (license terms and policies)
├── watermark.profile.json  (watermark configuration)
└── signature.json          (detached signatures)
```

**Deterministic tar:** Files in alphabetical order, no timestamps, no user/group, mode 644.

## 9. Identity Key Management

### 9.1 Key Generation

```bash
kdna identity init
```

Generates:
- `~/.kdna/identity/kdna.key` — Ed25519 private key (PEM, chmod 600)
- `~/.kdna/identity/kdna.pub` — Ed25519 public key (PEM)

### 9.2 Key Backup

```bash
kdna identity export --output kdna-identity-backup.age
```

Encrypts the private key with a user-provided passphrase (age encryption).

### 9.3 Key Rotation

```bash
kdna identity rotate
```

Generates new key pair. Old licenses are re-encrypted with new public key. Old private key is kept for 30 days then deleted.

## 10. Security Assumptions

1. **Registry is trusted** — it holds CKs and signs licenses. A compromised Registry can issue fake licenses. Mitigation: Registry private key in HSM.
2. **Buyer's private key is the user's responsibility** — if lost, licenses are unrecoverable. Mitigation: key backup passphrase.
3. **Plaintext exists in agent context** — any agent that uses KDNA has the plaintext in its context window. This is unavoidable. The defense is watermarking + licensing, not absolute prevention.
4. **Offline use is allowed** — `licensed` mode works offline for 7 days. This is a business decision, not a crypto limitation.

---

## 11. What This Protocol Does NOT Promise

- ❌ "No one can ever see the plaintext"
- ❌ "Copy-proof files"
- ❌ "Unbreakable encryption"
- ❌ "Replaces legal agreements"

What it DOES provide:
- ✅ Legitimate purchase mechanism with cryptographic proof
- ✅ Tamper-evident packages (signature verification)
- ✅ Leak accountability (watermark tracing)
- ✅ Managed revocation
- ✅ Clear separation of open / licensed / runtime modes
- ✅ Private key never leaves buyer's device
