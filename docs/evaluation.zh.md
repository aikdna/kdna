# KDNA 资产评估

> [English](./evaluation.md)

本文描述外部评价方法，不是 KDNA Core 格式有效性、官方质量分级或晋级权威。

## 先分开不同主张

评价必须说明自己测试的是哪一层：

1. **格式主张**：资产通过当前 Schema、完整性与加载合同；由 Core 和
   conformance tests 证明。
2. **交付主张**：预期资产版本经过授权、投影并交付给 Host；由 LoadPlan、
   Capsule 和 receipt 支持。
3. **遵循主张**：模型输出在具名任务和方法下与资产内容一致。
4. **因果主张**：相对于受控条件，加载资产造成了可测量的行为变化。
5. **价值主张**：某个评价者按具名 rubric 更偏好这种变化。
6. **结果主张**：资产使用对现实结果产生了贡献。

一层通过不能证明下一层。特别是，技术有效的 `.kdna` 不证明模型遵循、行为
改善、内容真实、作者专业或现实结果更好。

## Prompt、Skill 与 KDNA 对照

判断内容可能已经存在于 Prompt、Skill、Policy、RAG 语料或模型中。公平测试
必须说明每个条件实际收到了什么。

可用的对照包括：

- 没有等价判断内容的应用基线；
- 携带等价判断的最佳 Prompt；
- 携带等价判断的 Skill 或 Policy；
- 通过 `.kdna` Runtime Capsule 交付的相同内容；
- 表达不同判断系统的其他资产。

等价 Prompt 与 `.kdna` 产生相同行为，是完全有效的结果。它说明行为影响来自
内容而非载体；KDNA 仍可能在资产身份、完整性、授权、投影、替换和回滚方面
增加价值。

## 必须记录的评价坐标

每个公开结果都应记录：

- 评价者与评价目的；
- 精确资产身份、版本和 digest；
- 精确模型、Host、Prompt、Skill、工具和 Runtime 坐标；
- 任务与群体作用域；
- 对照条件和抽样方法；
- rubric、评分者与分歧处理；
- 时间、原始证据位置和已知限制。

不同评价者可以合理地得出不同结论。

## 示例方法

针对一个有边界的写作任务：

1. 在生成前定义判断主张与 rubric；
2. 冻结准确来源材料和模型设置；
3. 分别运行无等价判断基线、等价 Prompt 对照和 `.kdna` 条件；
4. 把资产 digest、LoadPlan 和 Capsule 的验证与输出评分分开；
5. 尽可能对输出顺序进行盲测；
6. 保留失败和无差异结果，而不只展示胜出案例；
7. 把行为发现写成评价者作用域内的证据，不写成资产内置属性。

## Core 可以报告什么

Core 可以报告 Schema、checksum、签名验证、授权状态、兼容、投影和 receipt 等
技术事实。它不能把评价分数变成内置的“优秀”“专业”“安全”“高质量”或
“推荐”状态。

外部目录和产品可以在自己的身份、方法、作用域和证据下发布这些评价。
