# 为什么需要 KDNA

通用模型已经拥有广泛知识，也具备有用的默认判断。但模型的默认判断，不一定是使用它的个人、创作者、专业者、团队或组织自己的判断。

KDNA 的意义，是让判断可以被明确表达、跨模型携带，并通过一致的运行时契约被 Agent 使用。

## 判断资产合同

事实、Memory、Skill、Workflow、Policy、Prompt 和 Eval 都可能携带判断。它们
拥有不同的主要合同，但没有哪个系统垄断某一种内容。

KDNA 把以下判断内容作为一份资产的主要责任：

- 什么最重要；
- 哪些区别不能混淆；
- 什么风险应该阻止行动；
- 选择时遵循什么价值和取舍；
- 对这个作者或团队而言，什么才算做好；
- 这些判断在什么情况下不适用。

KDNA 不替代数据治理、Memory、私有计算、RAG、评测基础设施、微调、Skill、
Prompt、Policy 或 Workflow。同一判断可以继续存在于任何载体中；只有当它需要
独立身份、完整性、授权、加载和生命周期时，KDNA 才增加价值。

## 谁的判断？

KDNA 不只属于企业，也不只属于获得认证的专家。

- 个人可以保存自己的偏好、价值、边界和选择方式；
- 创作者可以保存品味、表达标准和拒绝什么；
- 专业者可以封装诊断区别、风险阈值和取舍方法；
- 团队或组织可以把共同标准版本化，而不是把它锁进某个模型或应用；
- Agent 或工具也可以通过同一个公开协议创建 KDNA，作者身份不决定格式是否合法。

任何人都可以创建和发布 KDNA。KDNA Core 不判断其中的判断是否真实、正确、高级或有用。可选 Evidence 只描述一个资产已经观察到什么，不是创作许可证。

## 单资产与 Cluster

单个 KDNA 是原子、基础和默认路径：围绕一个清晰范围创建、校验、授权并加载一个判断资产。

KDNA Cluster 是显式的高级路径：多个资产可以被分配角色、路由、检查冲突，并围绕一个任务协作。Cluster 不替代单资产；单资产路径也不会暗中调用 Cluster Router。

## 开放创作，契约消费

开放协议不等于裸格式。兼容的 Agent 运行时遵循：

```text
inspect → LoadPlan → authorization → load/project → Runtime Capsule → Agent
```

直接解压 ZIP、解码 CBOR 或读取原始 payload 是开发者审计路径，不是兼容的 Agent 消费路径。正是这条边界，让同一个资产模型可以支持 public、licensed 和 remote 三种访问方式。

## “可验证”验证什么？

KDNA 可以验证格式、完整性、来源、授权状态和可选证据声明。它不验证某种判断是真理，也不证明作者值得信任。消费者根据自己的情境决定使用什么资产和证据。
