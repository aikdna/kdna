# 判断污染

## 定义

**判断污染**发生在 AI Agent 加载了与当前任务不匹配的 KDNA 领域时，导致 Agent 错误分类问题、应用错误优先级、使用错误风险模型、或给出错位建议——并且带着"已经加载了专家判断"的虚假信心。

## 与普通错误的区别

普通错误是 Agent 靠自己做出了错误判断，没有外部判断框架的干扰。判断污染不同：Agent 做出错误判断**是因为一个外部判断框架把它引导到了错误方向。** 加载的领域让输出变得比不加载更差。

## 为什么重要

KDNA 的价值主张是结构化领域判断能改善 Agent 决策。但这只在领域匹配任务时才成立。当它不匹配时：

- 团队管理领域应用到网站设计任务，导致 Agent 把视觉层级问题诊断为组织沟通失败
- 写作编辑领域应用到法律文件，导致 Agent 把可读性优先于法律精确性
- 销售领域应用到医疗咨询，导致 Agent 把患者顾虑当作需要克服的异议

每一种情况产生的输出都比不加载任何 KDNA 更差。

## 跳过优先

正因为判断污染的存在，KDNA 的运行时路由必须把"跳过"作为默认行为，而非例外。加载一个领域需要正向匹配证据。缺乏明确的匹配就是跳过的充分理由。

这不是 KDNA 的局限。这是它的特性。一个成熟的判断层必须知道什么时候该克制。

## 预防

判断污染通过 KDNA 运行时路由器预防（参见 [runtime-routing.md](./runtime-routing.md)）：

1. **意图门**：跳过根本不需要领域判断的任务
2. **先否定匹配**：先检查 `does_not_apply_when`，再检查 `applies_when`
3. **弱匹配 → 跳过**：当置信度低于阈值时，默认跳过
4. **信任门**：阻断信任验证失败的领域
5. **不自动安装**：永远不静默下载和加载领域
6. **全部追踪**：记录每一个路由决定以供审计

## 检测

判断污染可能已经发生的信号：

1. Agent 使用了不适合当前任务的领域特定术语
2. Agent 把一个领域的框架应用到不相关的问题上
3. Agent 的建议在错误领域内自洽，但对实际任务毫无意义
4. Agent 错过了显而易见的解决方案，因为它们不匹配加载的领域框架
5. 用户感觉 Agent "完全误解了整个情况"

## 报告

如果遇到判断污染：

1. 记录任务、加载的领域和受污染的输出
2. 检查领域的 `does_not_apply_when` 是否应该已经排除了这个情况
3. 如果 `does_not_apply_when` 缺失或不足，向领域作者提出添加建议
4. 如果路由逻辑失败，作为路由 bug 报告
