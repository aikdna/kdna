# AI Agent 不只需要工具：判断也需要可携带

判断已经存在于模型、Prompt、Skill、Policy、RAG、Memory、Workflow、文档和
人的经验中。KDNA 讨论的不是“谁拥有判断”，而是：什么时候一套判断需要成为
有独立身份、版本和加载合同的资产。

## 当前 Agent 栈

现代 AI Agent 的能力已经很全面了：

| 层 | 作用 | 例子 |
|----|------|------|
| 大模型 | 推理和生成 | GPT、Claude |
| 工具/MCP | 外部操作 | API 调用、文件系统、数据库 |
| RAG | 知识检索 | 文档问答、语义搜索 |
| 记忆 | 历史和连续性 | 会话历史、用户偏好 |
| Skills | 可重复工作流 | 模板、脚本、自动化 |

## 尚未统一的是资产合同

这些系统都可以参与“Agent 能做什么”和“Agent 如何判断”。它们不是互斥的
认知层，也不能保证 Agent 一定判断正确。

一个 Agent 可以调用 CRM、检索客户历史、生成邮件——但如果它不理解"客户第三次问价格"意味着什么，所有能力都在错误的方向上运行。

如果判断只存在于某一个应用的 Prompt 或 Skill 中，它仍然可以工作、测试和
版本化。但当这套判断需要跨模型、Agent 或应用移动时，各系统的身份、完整性、
授权、投影和生命周期合同通常不同。KDNA 处理的是这个可移植性问题。

## 症状

以下症状说明当前任务可能缺少合适的判断，而不是证明它必须使用 KDNA：

- 给用户太多信息，不知道什么更重要
- 能生成多个合理的方案，但不知道哪个更可靠
- 能执行任务，但不知道什么时候该停止、升级或拒绝
- 能模仿专家语言，但没有专家的判断结构
- 把讨论当成决策，把噪音当成信号，把偏好当成约束

## KDNA 增加什么

KDNA 不替代任何现有载体，也不垄断判断。它把被选择的判断系统变成规范资产：

```
判断内容
→ `.kdna` 资产身份与版本
→ 完整性和授权检查
→ LoadPlan 与投影
→ Runtime Capsule
→ 兼容 Agent / 应用
```

Prompt、Skill、RAG 或 Policy 仍可以表达完全相同的判断，甚至在具体任务中产生
相同行为。KDNA 的额外价值是跨兼容实现的资产合同，不是天然更深、更正确或
更聪明。

## 结论

当普通 Prompt、Skill、文档或知识库已经足够时，不必使用 KDNA。当判断需要
独立分发、授权、投影、撤销、替换或回滚时，KDNA 提供一种开放协议选择。
