# KDNA 的 Host 采用策略

> [English](./meta-cognition.md)
>
> 这是 Host 产品指导，不是 KDNA 格式规则，也不授权 Agent 扫描或选择机器上的
> 任意资产。

## 从用户授权开始，而不是从任务关键词开始

只有当用户明确选择了一份文件，或已经批准某个精确资产版本和 digest 在当前
工作区、应用或会话中使用时，Host 才能考虑该资产。

```text
用户选择或已批准附加
→ 可选资产集合
→ 当前任务适用性
→ LoadPlan 与授权
→ load / ask / skip / block
```

任务分类只能缩小已经授权的集合，不能反过来授予某份资产影响任务的权利。

## 四种决定

| 决定 | 含义 |
|---|---|
| `load` | 一份精确、已授权的资产明确适用，且 LoadPlan 允许加载 |
| `ask` | 多份候选或作用域有歧义，需要用户选择 |
| `skip` | 没有资产明确适用，或更简单机制已经足够 |
| `block` | 校验、完整性、授权、兼容或政策失败 |

资产不会因为版本更新、存在于全局目录、命中关键词或带有内容成熟度标签就自动
成为默认判断。

## 冲突顺序

Host 在使用时保持以下权威顺序：

1. 系统、安全、法律与工具权限；
2. 用户当前意图和明确任务约束；
3. 当前可验证事实与证据；
4. 所选资产的作用域、边界与判断；
5. Host 的展示与便利偏好。

两份已授权资产存在实质不同取舍时，应让用户选择；不要取平均，也不要由 Agent
静默选择自己偏好的一份。

## 可见与可控

Host 应让用户能够查看：

- 当前资产身份、版本和 digest；
- 附加范围：工作区、应用、会话或单次任务；
- 加载或跳过的原因；
- LoadPlan 和授权状态；
- 停用、切换与回滚入口。

普通回答不必朗读受保护的资产内部内容，但不能因此隐藏“哪份资产影响了任务”。

## 何时跳过 KDNA

Prompt、Skill、Policy、文档、Memory 或知识系统自己的合同已经足够时，继续使用
更简单的机制。任务超出作用域、当前事实推翻前提、用户拒绝资产框架，或 Host
无法保留其边界时，都应跳过 KDNA。

KDNA 提供独立资产和加载合同，不是每个判断任务的必选项。
