# KDNA 实战说明

> 以下是说明性示例，不是 benchmark 证据。它们解释判断资产如何参与任务，以及
> KDNA 合同究竟增加了什么。

## 影响行为的是内容，载体改变的是合同

假设销售系统采用这样一条判断：

> 在把“太贵”理解为降价请求之前，先区分真实预算限制与价值、风险或审批方面
> 的不确定性。

这条判断可以直接写进 Prompt、嵌入销售 Skill、放在 Policy 或手册中、从知识库
检索、被模型学习，也可以封装为 `.kdna`。如果同一模型收到等价内容，Prompt 和
KDNA 路径可能产生相同回答。这完全正常：影响行为的是判断内容。

KDNA 增加的是另一类声明：兼容 Runtime 可以识别检查了哪个资产版本和字节、
是否获得加载授权、投影了什么，以及交付了哪个 Runtime Capsule。它不证明这条
销售判断正确，也不证明它优于其他判断。

## 示例：会议决策状态

**输入：** 一段会议记录显示团队同意重写某个模块，但没有负责人、截止时间或
验收条件。

一套判断系统可能把它归类为“尚未形成可执行决定”，要求先明确责任；另一套判断
可能认为共识已经足以进入调研。KDNA 不裁决哪一种正确。它可以忠实封装其中
任何一套有作用域的判断，使 Host 知道自己加载了哪一份。

同一个任务可以同时使用：

- RAG 提供会议记录与当前项目证据；
- Memory 提供过去决定和用户偏好；
- Skill 提取决策记录；
- Policy 决定谁有权批准重写；
- KDNA 加载一份具名的决策状态判断资产；
- Evaluator 检查最终决策记录。

每个组件都可能包含判断，但它们的合同仍然不同。

## Prompt 与 KDNA 可以包含同一判断

| 关注点 | Prompt 系统 | KDNA 资产 |
|---|---|---|
| 可以表达结构化判断 | 可以 | 可以 |
| 可以版本化和测试 | 可以，由所属应用定义 | 可以，由 KDNA 资产合同定义 |
| 身份与 digest | 应用自行定义 | 标准 Manifest 与完整性合同 |
| 授权与投影 | 应用自行定义 | LoadPlan 与 Runtime Capsule 合同 |
| 工具权限 | Prompt 文本本身不授予 | 资产有效或加载本身不授予 |
| 内容质量 | 需要外部评价 | 需要外部评价 |

只有当判断需要脱离某个 Prompt、Skill、模型或应用独立流转时，KDNA 才增加
价值。没有这个需求时，更简单的载体可能更合适。

## 技术示例能证明什么

它可以证明：

1. `.kdna` 符合当前格式；
2. 身份与字节通过完整性检查；
3. 兼容 Runtime 生成了 LoadPlan 和 Runtime Capsule；
4. Host 收到了被记录的投影。

它不能在缺少独立证据时证明：

- 资产判断真实、专业、真诚或值得采用；
- KDNA 比等价 Prompt 或 Skill 产生更好回答；
- 模型遵循了每一项投影判断；
- Agent 获得了采取外部行动的权限。

这种分离是有意的：KDNA 标准化判断资产交付，内容和结果仍由作者、Host、
Evaluator 与用户负责。
