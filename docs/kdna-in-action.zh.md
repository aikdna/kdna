# KDNA 实战示例

跨领域的前后对比案例，展示 KDNA 如何改变 Agent 的判断。

## 销售：价格异议

**用户输入**："你们的报价太贵了，能不能便宜点？"

**无 KDNA 的回应**：
> 理解您的顾虑。我们确实比竞品贵一些，但我们提供更多的功能。要不要我给您发个详细的对比表格？

这是一个合理的销售话术——但它没有诊断用户的真实顾虑。

**有 KDNA 的回应**：
> 在讨论价格之前，我想先了解您最关心的是什么——是预算限制，还是对我们产品能否达到您预期效果的不确定？

KDNA 的公理"价格异议是确定性不足，不是价格问题"改变了 Agent 从"对比功能"转向"诊断顾虑"。

## 管理：执行延迟

**用户输入**："团队又延期了，我需要跟每个人谈谈他们的工作态度。"

**无 KDNA 的回应**：
> 建议安排一对一谈话，了解每个人的困难，设定明确的绩效目标。

这是标准的管理建议。

**有 KDNA 的回应**：
> 在跟个人谈话之前，先诊断系统：大家的期望是否清晰？资源和权限是否到位？有没有早期反馈机制能在延迟扩大前就发现？

KDNA 的公理"执行失败是系统输出，不是动力问题"改变了 Agent 从"管人"转向"诊断系统"。

## 会议：虚假决策

**用户输入**：一段会议记录，显示团队"达成共识"要重构某个模块。

**无 KDNA 的回应**：
> 会议决定重构认证模块。建议下一步：分配开发任务，设定 Q3 时间节点。

**有 KDNA 的回应**：
> 会议讨论了重构，但缺少三个关键决策要素：没有指定负责人、没有明确时间节点、没有定义成功标准。当前状态：UNRESOLVED。建议：首先指定负责人，由其提出包含时间线和成功标准的方案，再进行决策。

KDNA 的决策状态判断框架阻止了"把讨论当决策"的错误——这在实际工作中是最常见的生产力陷阱之一。

## 关键差异

| 维度 | 无 KDNA | 有 KDNA |
|------|---------|---------|
| 销售 | 对比功能 | 诊断顾虑来源 |
| 管理 | 管个人 | 诊断系统 |
| 会议 | 总结说了什么 | 判断是否形成了决策 |
| 所有场景 | 基于通用知识回应 | 基于领域判断框架回应 |
