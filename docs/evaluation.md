# Assessing a KDNA Asset

> [中文版](./evaluation.zh.md)

Assessment belongs to the person or product making a claim. It is not KDNA
Core format validity, an official quality score, or a condition for creating a
valid asset.

Keep these facts separate:

1. **Format validity** — the container and Schema are valid.
2. **Integrity and provenance** — bytes, digests, and declared origins agree.
3. **Authorization and delivery** — the exact asset was allowed, projected,
   and delivered to a named Host.
4. **Adoption** — observed output was consistent with selected judgment under a
   named task and method.
5. **External assessment** — a named evaluator preferred, rejected, or was
   indifferent to the result under a stated rubric.
6. **Outcome** — a real-world result was observed in a defined setting.

One fact never proves the next. In particular, Core must not turn an external
assessment into built-in “good,” “expert,” “safe,” or “recommended” metadata.

Any public claim should identify the claimant, exact asset version and digest,
Host and runtime coordinates, scope, method, evidence location, and known
limitations. Different people can legitimately judge the same asset
differently.
