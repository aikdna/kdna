# KDNA 运行时适用性策略

> 中文说明。权威英文边界见 [runtime-routing.md](./runtime-routing.md)。

KDNA Core 不替用户或 Agent 决定“电脑上哪份资产应该影响当前任务”。资产选择
与任务适用性属于 Host 政策，不属于 `.kdna` 格式权威。

## 输入边界

Host 只能从以下集合开始：

1. 用户为当前操作明确选择的一份 `.kdna` 文件；或
2. Host 已经记录用户批准、并以身份、版本和 digest 固定的精确附加项。

扫描目录、发现某个文件、把文件保存进缓存、安装 Skill 或根据任务关键词猜测，
都不产生授权，也不能把资产加入当前任务的候选集合。

## Host 可以做什么

在已经批准的候选集合内，Host 可以根据资产声明的 `applies_when`、
`does_not_apply_when`、当前任务和自身政策返回：

- `load`：适用且授权、完整性和兼容性检查通过；
- `ask`：多个已批准资产可能适用，选择会改变判断方向；
- `skip`：本任务不适用或不需要加载；
- `block`：资产原本适用，但授权、完整性、兼容性或安全条件失败。

Host 不能扫描全局资产库后自主选择，不能把“可能相关”解释成用户同意，也不能
把相互冲突的资产静默混合。

## 可见性和控制

加载发生时，Host 必须让用户能够查看：

- 资产身份、版本和 digest；
- 生效作用域与本次采用原因；
- 授权状态和投影是否完整；
- 停用、切换和回滚入口。

界面不必在每段回答中重复 KDNA，但不能隐藏 KDNA 是否影响了当前任务。

## 命令边界

`kdna plan-load <file>` 与 `kdna load <file>` 处理一份精确资产。已发布 CLI 中的
`route`、本地 Store、发现和多资产表面是待重新认证的高级实现，不是默认用户
路径，也不是 Agent 自主取得选择权的依据。
