# 什么时候使用 KDNA

不是每一种判断都需要 KDNA。触发条件不是“判断还是流程”，也不是“KDNA 还是
Skill”。Prompt、Skill、Policy、Workflow、Memory、RAG 和知识库都可以携带
判断。

## 当资产合同很重要时使用 KDNA

同时存在多项下列需求时再使用 KDNA：

- 判断需要独立于某个 Prompt、Skill、模型或应用获得身份；
- 必须验证准确版本与字节完整性；
- 加载需要授权或按任务投影；
- 同一资产必须跨兼容 Runtime 移动；
- 替换、撤销、回滚或消费证据很重要；
- 判断与使用它的能力需要独立演进。

例如：由多个写作工具共同使用的品牌判断资产；只授权给某个团队的审查标准；
或必须独立于 Agent Skill 固定版本和回滚的诊断资产。

## 现有载体足够时使用更简单的机制

如果 Prompt、Skill、文档、Policy、Memory 或检索系统现有的身份、测试、部署、
授权与更新路径已经满足要求，就把判断留在那里。

常规格式化也可能包含安全和正确性判断；复杂诊断也可能由经过测试的 Prompt
很好地完成。任务复杂度本身不能决定是否需要 KDNA。

## 边界问题

问自己：

> 这套判断是否需要作为独立资产被管理和加载，还是当前载体已经足够？

前者使用 KDNA，后者使用更简单的现有机制。
