# KDNA 集群

KDNA Cluster 是一个可组合的判断系统：多份有边界的 KDNA 领域资产，在明确角色和路由策略下协同工作，处理复杂任务。

Cluster 不是一个更大的合并版 `.kdna`，也不是把多份 `.kdna` 合并成一个文件。每份领域资产仍保留自己的范围、版本、来源和评测边界；Cluster 只定义哪些资产参与、何时加载、冲突如何暴露，以及组合后的系统如何被审查。

## 为什么不能做一个大 KDNA？

一个覆盖宽泛领域的大 KDNA 有四个问题：

| 问题 | 例子 |
|------|------|
| **太笼统** | 一个包含 20 条原则的"商业增长 KDNA"会变成一堆正确的废话 |
| **上下文浪费** | 每次加载全部判断模式，但当前任务只需要其中 2 个 |
| **框架冲突** | "速度优先"和"信任优先"在同一个资产里无声冲突 |
| **无法测试** | 所有东西打包在一起，无法证明哪个具体判断改善了 |

## Domain KDNA vs Cluster vs Work Pack

| 层级 | 作用 | 回答的问题 |
|---|---|---|
| **Domain KDNA** | 一份打包为 `.kdna` 的领域判断资产 | 在这个领域和任务中，AI 应按什么标准判断？ |
| **KDNA Cluster** | 多份领域资产之上的组合层 | 哪些 KDNA 参与？各自是什么角色？按什么策略加载？ |
| **Work Pack** | 判断 + 执行材料组成的可复用工作能力 | 一类工作应如何执行、审查和改进？ |

Work Pack 可以包含 KDNA Cluster、Skill、任务模板、输出模板、审查门槛、风险策略、推荐工具和 trace/feedback 契约。KDNA 仍然是其中的判断层。

## 集群角色

集群中的每份领域资产有四种角色之一：

| 角色 | 职责 | 规则 |
|------|------|------|
| **Primary（主）** | 当前任务的主判断镜头 | 每次任务只有一个 |
| **Advisor（辅）** | 从另一个角度补充判断 | 每次任务最多 3 个 |
| **Constraint（约束）** | 硬边界，可以覆盖不安全建议 | 检测到风险时加载 |
| **Critic（审查）** | 反向审计主判断的输出 | 审查模式下加载 |

## 组合规则

1. **必须有主判断。** 每次任务有且只有一个 Primary KDNA。
2. **不能平均。** 两个资产意见冲突时，暴露冲突——不要折中成"既要又要"的废话。
3. **冲突必须可见。** 告诉用户："领域 A 认为这是 X。领域 B 认为这是 Y。哪个视角更合适？"
4. **按任务阶段加载。** 诊断→设计→表达→审查，每个阶段加载不同资产。不要一次性全加载。
5. **保留来源归属。** 多个领域影响输出时，trace 必须保留哪条判断来自哪个领域。
6. **集群有自己的评测。** 不只测试单个资产，还测试集群作为一个系统是否选对了主判断、避免了无关资产、识别了冲突、输出了分层判断。

## 集群 Manifest

集群由 `kdna.cluster.json` 定义，结构与单个资产的 `kdna.json` 不同：它描述的是多份领域资产之间的组合关系。

## 什么时候用集群而不是单个领域资产

| 用单个领域资产 | 用集群 |
|--------|--------|
| 任务只有一个清晰的判断分岔点 | 任务跨越多个判断维度 |
| 一个专家视角就够 | 不同阶段需要不同视角 |
| 领域窄且边界清晰 | 领域宽且有交互的子领域 |
| 你在创建第一个 KDNA | 你有多个已验证的领域资产 |

## 与现有概念的关系

- **判断模式**：最小单元——信号/误读/框架/边界（定义在 `benchmarks/`）
- **KDNA Asset**：包含 2-6 个标准 KDNA 判断文件的领域资产（定义在 `SPEC.md`）
- **KDNA Cluster**：有明确角色和路由策略的可组合领域资产系统（本文档）
- **Work Pack**：把 KDNA 或 KDNA Cluster 与技能、模板、审查门槛、trace/feedback 契约组合成可复用工作能力

这些层级互补：模式在领域资产里面，领域资产被组织成集群，集群可以进入 Work Pack。
