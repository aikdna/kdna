# KDNA：AI Agent 领域认知的开放格式

*让领域判断显式、可移植、可验证、可复用*

**白皮书 v1.0 — 2026 年 5 月**

---

## 摘要

AI Agent 正在变得足够强大，可以搜索、推理、生成、调用工具并执行多步骤工作。随着这种能力增长，一个更深层的问题浮现了：不仅是 Agent 能做什么，而是它如何判断什么值得做、什么该拒绝、怎样才算做好——在一个具体领域中。

如今，领域判断已经存在。它出现在专家习惯、团队约定、提示词、风格指南、评分标准、检查清单、检索文档、工作流规则、测试、微调行为和工具策略中。这些机制可以也应该继续被使用。

但领域判断通常碎片化地分散其中：一部分写在提示词里，一部分藏在文档里，一部分嵌入在工具里，一部分内化在模型权重里，一部分仅存于经验丰富的从业者脑中。当 Agent 失败时，很难检查是哪条判断规则缺失、过时或被忽略了。

KDNA 是一种表示领域判断的开放格式。它将构成专家判断的原则、概念边界、误解模式、场景信号、推理链和自检规则编码为 AI Agent 可加载、人类可检查、工具可验证的形式。它的贡献不是声称判断从未在 AI 系统中存在过，而是判断现在值得拥有自己的、可移植、可验证的表示。

规范是开放的，参考工具链以 Apache 2.0 发布，一个早期公开域注册表已可用。KDNA 不替代提示词、知识库、Skills、MCP、工具、检索、评估或微调。它给它们一个更清晰的判断参照。

## 1. 为什么领域判断需要一种格式

### 1.1 从能力到判断

AI Agent 正从被动响应走向主动工作。它们可以编写代码、编辑文档、调用 API、操作工具、综合研究、执行工作流。这种转变改变了核心问题。

在早期 AI 使用中，主要关注点常常是：模型能产生有用的答案吗？在 Agent 式 AI 中，问题变成了：Agent 如何决定什么值得做、什么该避免、怎样才算做好？

一个编码 Agent 可能能够修改文件和运行测试。一个写作 Agent 可能能够生成流畅的文本。但在每种情况下，更难的问题是领域特定的：什么让一个修复彻底而非表面？什么让一个论证有实质而非装饰？什么让一个产品决策在战略上连贯而非仅仅合理？

随着 Agent 能力增长，判断力变得更加重要，而非更不重要。

### 1.2 判断已经存在，但是碎片化的

领域判断并非缺失。专家已经在使用它。团队已经在依赖它。AI 系统已经通过许多现有机制近似了它的部分：系统提示词、风格指南、代码审查标准、合规策略、内部手册、检查清单、评估套件、微调行为和工具验证器。

这些机制是有用的。KDNA 不反对它们。

问题是判断散落在其中。一条原则可能存在于提示词里。一个概念边界可能被描述在风格指南里。一种失败模式可能只有一位资深审查员知道。一项检查可能实现在工具里。一条首选推理模式可能从未被写下过。当 Agent 失败时，很难检查是哪条规则缺失、过时或互相矛盾。

这不是提示词问题。这是表示问题。

### 1.3 表示问题

如果领域判断仅存在于提示词中，它通常是脆弱的和任务特定的。一个有效的提示词在某个会话中可能有用，但难以跨系统验证、版本化、复用和组合。

如果仅存在于文档中，Agent 可能检索到它，但不会持续应用。文档可以包含原则，但它本身并不区分公理与示例、必需的自检与背景解释。

如果仅存在于模型权重中，它可能很强大但不透明。行为难以检查、精确更新，或归因到某条具体原则。

如果仅存在于人脑中，它容易丢失。专家离职时，判断随之而去。团队扩张时，判断变得不一致。

KDNA 通过将领域判断视为一等制品来解决这个问题：某种可以被编写、评审、验证、版本化、分发、加载和改进的东西。

### 1.4 KDNA 加了什么

KDNA 不声称现有方法是错的。它声称随着 Agent 变得更加主动，这些方法受益于一种可移植的判断表示：一种可被检查、验证、复用和跨系统组合的表示。提示词定义当前任务。知识库提供信息。Skills 封装流程。微调塑造模型行为。评估衡量性能。KDNA 表示应指导 Agent 如何使用这些能力的领域判断。

KDNA 在精神上接近规则文件、评分标准、风格指南和结构化提示词。它的贡献不是原则从未被写过。而是 KDNA 为领域判断定义了一种通用的开放包格式、验证规则、加载语义和组合模型。

## 2. KDNA 是什么

### 2.1 定义

KDNA 是一种开放的、基于文件的格式，将领域判断编码为结构化、可验证的包。一个 KDNA 域是一个包含少量标准 JSON 文件的目录。它们共同描述指导特定领域判断的原则、概念、边界、误解、场景、推理模式和自检规则。

一个域不试图存储所有信息。不试图替代专家。它捕获专家在做决定时反复使用的判断结构：什么重要、什么该拒绝、必须保留什么区分、常见什么错误、什么信号应改变策略、遵循什么推理路径、最终确定输出前必须检查什么。

KDNA 是刻意紧凑的。一个有用的域可能只包含两个文件。更丰富的域最多包含六个。目标不是体量——是清晰、特异和可复用。

### 2.2 KDNA 编码什么

**公理**——不可违反的原则。不是鼓舞人心的陈述。是判断锚点，定义 Agent 推理时必须保持为真的东西。示例：*Bug 修复应解决根因而非仅消除可见症状。*

**本体**——核心概念及其边界。一个概念是什么，它不是什么，哪些区分不能被模糊。示例：*情感共鸣帮助读者识别真实体验。焦虑贩卖通过让读者感到不足来施加压力。*

**框架**——决策方法，带有触发条件、步骤和预期输出。

**立场**——默认态度。在具体任务开始前，领域倾向于偏好或拒绝的东西。示例：*偏好具体张力而非装饰性形容词。*

**术语**——带操作定义的标准术语，以及带具体替代方案的禁用词。词语常常承载隐藏的预设。一个域可以拒绝某些术语，不是因为风格原因，而是因为它们编码了错误的心理模型。

**误解**——常见的错误理解，附带正确理解和必须保留的关键区分。帮助 Agent 检测输出是否滑向已知的错误模式。

**自检**——Agent 在最终确定输出前必须用是/否回答的问题。可验证，不空泛。示例：*建议是否识别了权衡，而非仅仅呈现好处？*

**场景信号**——应改变 Agent 响应方式的可观察输入模式。

**推理链**——从前提→结论→实践后果的结构化路径。

**能力层级**——"进步"的定义，使能力可以随着时间被评估。

### 2.3 域结构

一个 KDNA 域是一个以蛇形小写命名的目录。完整域最多包含六个标准文件：

| 文件 | 编码内容 | 必需 |
|------|---------|:----:|
| `KDNA_Core.json` | 公理、本体、框架、核心结构、立场 | 是 |
| `KDNA_Patterns.json` | 术语、误解、自检 | 是 |
| `KDNA_Scenarios.json` | 触发策略变化的场景信号 | 否 |
| `KDNA_Cases.json` | 展示判断力的具体案例 | 否 |
| `KDNA_Reasoning.json` | 从前提→后果的推理链 | 否 |
| `KDNA_Evolution.json` | 能力层级与测量 | 否 |

最小有效域包含 `KDNA_Core.json` 和 `KDNA_Patterns.json`。每个文件包含一个必需的 `meta` 对象，含 `version`、`domain`、`created`、`purpose` 和 `load_condition`。所有文件都根据已发布的 JSON Schema 验证。

规范形式是 JSON 文件目录——源文件优先、透明、可版本控制。为便于分发，工具链可将域打包为 `.kdna` 容器（基于 ZIP 的单文件，含打包元数据）。

### 2.4 为什么文件格式很重要

一种文件格式与模型、Agent 框架或平台相比，可能显得低调。但格式之所以重要，是因为它使事物变得可移植。文档之所以可移植，是因为有文档格式。包之所以可复用，是因为生态系统有包格式。API 之所以可互操作，是因为有接口描述。配置之所以可管理，是因为行为可以与代码分离。

KDNA 将这一逻辑应用于领域判断。正如配置文件将环境特定行为与应用代码分离，KDNA 将领域判断与临时提示词和 Agent 实现分离。它给了判断一个可以安放的地方。

### 2.5 KDNA 不是什么

| KDNA 不是 | 原因 |
|-----------|------|
| **不是提示词库** | 提示词是任务范围的。KDNA 是领域范围且版本化的。 |
| **不是知识库** | 知识库存储信息。KDNA 编码判断约束。 |
| **不是工作流引擎** | 工作流定义步骤。KDNA 定义跨步骤的判断。 |
| **不是微调模型** | 微调将行为内化在模型权重中。KDNA 将选定的判断原则保持为显式且可审计。 |
| **不是 Agent 框架** | KDNA 是格式——不是 Agent。任何框架都可以实现加载。 |
| **不是正确性保证** | KDNA 提供显式参照。质量取决于编码其中的判断。 |

## 3. KDNA 如何工作

### 3.1 验证

KDNA 域可以在多个层次上被验证：

- **结构 lint**——必填文件存在、字段已填写、ID 唯一、自检可用是/否回答、无禁用模式、标记可能空泛的公理或不可操作的自检。
- **Schema 验证**——字段匹配 JSON Schema 类型，数组有必需项，跨文件引用一致。
- **质量评估**——针对人工标注测试用例的 benchmark 评分，衡量 Agent 输出是否遵循领域原则。

前两个层次已完全自动化。第三个层次正在萌芽。Schema 合规不保证质量。一个域可以在结构上有效但判断力很弱。KDNA 将结构有效性与判断质量分开。

### 3.2 加载

当 Agent 加载 KDNA 域时，loader 读取并验证域文件，使用标准模板渲染为结构化上下文块，并使该上下文在运行时对 Agent 可用。渲染后的上下文将域结构保留为独立命名的段落。多个域可同时加载——每个贡献独立的判断参照。

### 3.3 组合与集群

丰富的判断力常常来自组合多个较小的域。一个编码 Agent 可能需要 Bug 诊断判断、TDD 判断、安全审查判断和技术写作判断。每个应保持独立。

**集群**将多个域分组并声明组合方式：
- **Primary 域**始终加载。
- **Secondary 域**仅当场景信号匹配时加载。
- **边界保留**——每个域的公理、术语和自检保持独立。当域冲突时，Agent 汇报冲突，而非静默地融合互不兼容的原则。

### 3.4 示例

**写作。** AI Agent 被要求写产品公告，可能产生流畅的文案。但缺乏显式判断标准时，它可能使用焦虑贩卖式语言，用听上去深刻却不改变理解的短语装饰文本，或未能为读者创造一个真正的决策问题。写作 KDNA 域可编码这些区分：关于结构性论证的公理，抽象赞美而无具体含义、或声称转变却未指定前后具体变化等禁用模式，"核心张力能否用一句具体的话陈述？"等自检。Agent 遵照一个已声明的编辑判断结构工作。

**代码审查。** 审查 PR 的 Agent 可能有访问权限、测试、linter 和长篇提示词。但它可能批准一个消除症状却保留根因的修复——因为没有判断规则显式要求根因分析。代码审查域可编码"消除症状却保留失败条件的修复是不完整的"等公理，"将测试通过混同于风险移除"等误解，"变更是否引入了新的隐藏依赖？"等自检。

## 4. KDNA 在哪里

KDNA 与现有 Agent 组件并排时最容易理解：

| 组件 | 主要角色 | 通常未标准化什么 |
|------|---------|----------------|
| **提示词** | 定义当前任务或行为 | 作为版本化制品的可复用领域判断 |
| **知识库** | 提供信息和参考 | 结构化判断约束、自检、组合规则 |
| **Skills** | 封装可复用流程 | 接受或拒绝结果的领域原则 |
| **MCP / API** | 连接 Agent 到工具和数据 | 领域特定的评估原则 |
| **微调** | 内化行为模式 | 可检查和精确编辑的判断规则 |
| **评估套件** | 衡量性能 | 被测量的可移植判断包 |
| **KDNA** | 表示领域判断 | 不执行任务，不替代模型能力 |

KDNA 不是这些组件的竞争者。它是补充。提示词告诉 Agent 现在做什么。Skill 告诉它如何执行流程。MCP 连接工具。知识库提供参考。微调塑造行为。评估衡量结果。KDNA 给 Agent 一个结构化的领域判断参照，用于决定什么重要、什么该拒绝、怎样才算做好。

## 5. 为什么显式领域判断很重要

### 5.1 可检查性

当判断隐藏在提示词、模型行为或未记录的专家习惯中时，它难以被检查。KDNA 使判断可读。人可以打开一个域并问：这些原则正确吗？概念边界清晰吗？自检具体吗？Agent 的输出可能仍需要审阅，但引导它的判断参照不再不可见。

### 5.2 可移植性

一个提示词在一个工具中有效，在另一个中未必。一个团队约定在某个部门中存在，在另一个中未必。微调行为绑定于某个模型。KDNA 使领域判断可移植——存储在仓库中，打包为 `.kdna` 容器，通过工具安装，由不同 Agent 加载，随时间版本化。

### 5.3 可验证性

非结构化的判断难以检查。KDNA 引入多个验证层次：Schema 验证、结构 lint、跨文件一致性、自检质量规则、benchmark 评估、人工审阅和版本历史。这不使判断自动正确。但使判断可被审阅。一个域可以被比较、改进，并根据证据获得质量徽章。

### 5.4 可组合性

真实工作是跨领域的。一个任务可能涉及技术准确性、战略判断、受众理解、法律谨慎和品牌调性。如果全部合并到一个提示词中，结果变得脆弱。KDNA 允许域在加载时保持独立，每个贡献独立的判断参照。

### 5.5 可传承性

组织常常低估其真正专业知识所在。它在经验丰富者的反复判断中：他们首先注意到什么、他们立即拒绝什么、他们考虑哪些权衡、他们学会避免哪些错误。当这些人离开时，大部分判断随之而去。KDNA 不完全保存一个人的专业知识——没有格式能做到。但它可以保存反复出现的判断原则、概念边界、失败模式和自检规则。这足以使专家判断更可教、更可审阅、对 AI Agent 更可用。

## 6. 公共和私有的域

KDNA 支持两种模式：

**公共域**在明确的内容许可（如 CC-BY-4.0 或其他开放许可）下通过注册表公开共享。它们作为参考实现、学习资源、社区维护的判断包和高质量编写的示例。

**私有域**编码专有或组织特定的判断：内部设计审查标准、公司品牌调性判断、合规解释框架、资深架构师的审查原则。可存储在私有仓库或内部注册表中。

格式和工具对两者完全相同。只有分发渠道不同。

## 7. 局限与风险

一种严肃的格式必须定义其局限。

**并非所有专业知识都可被充分形式化。** 有些判断是深度隐性的——取决于感知、情境、时机或多年实践。当领域具有反复出现的原理、稳定的概念边界、已知的失败模式和可审阅的输出标准时，KDNA 最有用。

**显式判断可能僵化。** 写得不好的域可能过度约束 Agent、拒绝有用的变体或保留过时的假设。域需要版本化、评审、反馈和演化。

**Schema 有效性不等于质量。** 一个域可以通过验证但仍然很差——空泛的原则、肤浅的区分、薄弱的自检。工具检查结构。质量需要专家评审和真实使用评估。

**上下文限制不可忽视。** 加载过多域可能稀释而非改进判断。集群必须精挑细选。更好的选取比更多的 KDNA 更好。

**域可能冲突。** 品牌域可能鼓励情感强度，而合规域要求保守措辞。KDNA 应暴露而非隐藏这些冲突。冲突汇报是负责任的判断组合的一部分。

**KDNA 不替代问责。** 即使有了 KDNA，人类仍需为高风险决策负责。KDNA 提高了可检查性。它不消除责任。

## 8. 生态

### 8.1 开放格式与工具链

KDNA 由开放协议规范（SPEC v1.0-rc）和覆盖所有标准域文件的 JSON Schema 定义。参考实现和工具链以 Apache 2.0 发布。任何能读取结构化文件并注入上下文的 Agent 或框架均可实现 KDNA 加载。

| 工具 | 用途 | 状态 |
|------|------|:----:|
| `@aikdna/kdna-core` | 纯逻辑库（零依赖）：加载、验证、lint、渲染、组合域 | 已发布（npm, v0.2.2） |
| `@aikdna/kdna` CLI | 命令行工具：validate、verify、pack、unpack、install、compare、match、setup | 已发布（npm, v0.7.0） |
| `@aikdna/agent` | TypeScript SDK，含 KDNAAgent 类用于 Agent 集成 | 已发布（npm, v0.1.1） |
| kdna-vscode | VS Code 扩展：编辑器内验证、预览、创建域 | Marketplace 已发布 |
| kdna-website | aikdna.com：域浏览器、查看器、文档、工具、工作室 | 已部署 |

所有工具共享同一核心库，确保一致行为。

### 8.2 注册表

公开的 KDNA 注册表（`domains.json`）是可用域的机器可读索引。目前包含 9 个开放域，按双轴分类：**领域范围**（何种场景）和**判断模式**（何种判断类型）。注册表条目携带元数据：名称、作者、版本、状态、关键词标签、质量徽章及可用的评估历史。注册表处于早期，正在增长。

### 8.3 质量信号

KDNA 域不应仅因通过 Schema 验证就被视为同等价值。质量需要额外信号：专家评审、benchmark 性能、使用证据、版本历史、社区反馈和清晰的归属。格式保证结构。它不保证专业能力。

## 9. 经济愿景

### 9.1 判断力作为可复用资产

互联网使信息充裕。AI 使生成充裕。但判断力仍然稀缺——不是因为它无法被表达，而是因为它往往未被结构化为可复用资产。一个 KDNA 域可以编码一个人、团队或社区在领域内如何判断。如果有用，它可以跨任务、Agent、团队和组织被复用。这赋予了领域判断知识产权的某些属性：归属、版本化、评审、许可、声誉和随时间的改进。

### 9.2 创作者域

一位受尊重的编辑可以发布写作判断域。一位资深工程师可以发布代码审查域。一位产品负责人可以发布优先级判断域。这些不会是模板——它们是承载创作者原则、区分和自检的结构化判断包。随着注册表和评估系统成熟，高质量域可积累声誉。如果域在真实工作流中证明有用，构建了经过验证、良好评审的域的创作者，其价值可能跨作品积累。

### 9.3 企业私有域

组织持有自己很少识别为资产的认知资产：资深工程师如何审查架构，合规团队如何解读风险，产品团队如何权衡优先级。这些判断模式散落在人、文档、工作流和评审评论中。私有 KDNA 域让组织以 AI Agent 可加载、人类可审查的格式编码这些专业知识。这不是专业知识的自动化——是专业知识的保存、分发和可操作化。

### 9.4 长期演进

1. **格式层**（今天）——标准化的领域判断编码。
2. **验证层**（萌芽）——Benchmark 结果、评审流程、质量徽章。
3. **发现层**（建设中）——注册表、面状浏览、域预览。
4. **市场层**（未来）——许可、归属、收入分成。
5. **资产层**（远方）——认知资产作为被认可的可复用知识产权类别。

我们处于 L1-L2，有意识地建设。我们不把 KDNA 描述为金融工具。

## 结语

AI Agent 正在变得更强大、更互联、更主动。这种进步不使人类判断变得不重要。它使人类判断的表示变得更加重要。

领域判断已经存在于许多地方：提示词、文档、工具、策略、工作流、评估、模型行为和专家习惯中。但它往往是碎片化的、隐式的、任务范围绑定的、平台依赖的，且难以验证或组合。

KDNA 解决了这个表示问题。它赋予领域判断一种开放格式——一种可以被编写、验证、加载、组合、版本化、共享和改进的格式。它不替代现有机制。它给它们一个更清晰的判断参照。

项目处于早期。格式和工具链已经存在。下一个挑战是通过高质量域和真实使用来证明——显式领域判断能够以可衡量、可检查和可复现的方式改进 Agent 行为。
