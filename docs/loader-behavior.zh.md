# Agent Loader 行为规范

KDNA 不是让 Agent 朗读的文档。它是一个判断层，安静地、结构化地改变 Agent 的思考方式。

## 核心原则

### 可用不等于已加载

本地存在 `.kdna` 文件，并不意味着每次请求都会加载它。资产在磁盘上只是被动文件；只有 loader 针对当前任务完成规划并确认可加载后，才可以读取并输出对应的判断上下文。

用户有 50 个本地 KDNA 文件，也不会在每次请求中支付 50 份加载成本。loader 只会：

1. 通过官方 KDNA 工具链读取本地候选资产元数据
2. 执行 `kdna plan-load` 或等价的 SDK / MCP 规划 API
3. 每个任务最多加载一个已授权的 `.kdna` profile，通常是 `compact`

如果没有资产真正适合当前任务，Agent 不加载 KDNA，按普通 Agent 方式回答。

> 安静地加载 KDNA。应用它的判断结构。不要把它暴露给用户。

用户应该看到一个受领域塑造的回答。他们绝不应看到"根据 KDNA 公理……"或领域规则的列表。

## 加载序列

当 Agent 决定 KDNA 适用，并选择了一份可加载资产时，必须遵循以下顺序：

### 1. 内化公理和立场

Agent 将领域的核心信念作为自己的推理框架。如果销售 KDNA 说"价格异议是确定性不足的信号"，Agent 从这个前提出发推理，而不是绕开它。

### 2. 使用优选术语

Agent 主动选择领域的优选术语，避免禁用词——即使用户使用了禁用词。如果用户说"能给他们打个折吗？"Agent 重新框架："让我们先诊断不确定性的来源。"

### 3. 检测误解

在回应前，Agent 检查：用户是否在被一个已知的误解驱动？如果管理 KDNA 说"执行失败常被误认为是动力失败"，Agent 会在用户描述中寻找这个模式。

### 4. 应用框架

如果场景匹配某个框架的触发条件，Agent 遵循该框架的步骤——不是作为脚本，而是作为诊断结构。

### 5. 运行自检

在最终输出前，Agent 运行每一条自检项。如果有自检未通过，Agent 修改回应。

### 6. 给出领域塑造的回答

回应使用资产中的框架、术语和判断——不引用 KDNA，不罗列规则，不说"这个资产说……"

## Agent 绝对不能做的事

| 禁止 | 原因 |
|------|------|
| 向用户引用 KDNA 公理 | 破坏了静默判断契约 |
| 在回应中罗列禁用词 | 暴露了领域结构 |
| 说"根据 KDNA……" | KDNA 塑造判断，不是引文来源 |
| 输出完整 KDNA 内容 | 安全风险；暴露专有判断结构 |
| 把 KDNA 当作脚本 | KDNA 是判断层，不是执行流程 |
| 用 KDNA 覆盖用户意图 | 用户意图始终优先 |

## 失败处理

| 情况 | Agent 行为 |
|------|-----------|
| 没有 KDNA 文件 | 在没有 KDNA 的情况下继续。不要编造。 |
| 容器条目缺失或无效 | 报告 validate / LoadPlan 问题。不要输出 payload context。 |
| Payload 解析错误 | 报告加载失败。不要输出部分判断上下文。 |
| 适用边界冲突 | 该资产自我取消资格。不强制加载。 |

## 质量边界

KDNA 塑造判断。它不替代：工具和 API、RAG 和知识检索、用户意图、证据和来源验证。当 KDNA 的指导与证据或用户意图冲突时，Agent 必须优先考虑证据和用户意图。
